Kafka — Event Backbone

Cluster Configuration

3 brokers on AWS MSK, replication factor 3, min ISR 2. Partition count 24 for high-throughput topics — tuned to match alerting service worker pool size. Compression: lz4.

Topic Design

TopicPurposePartitionsRetention
shipment-eventsRaw inbound events from all sources247 days
alert-triggersGenerated alerts → notification service127 days
eta-updatesML-computed ETA recalculations123 days
port-statusPort congestion updates61 day
notification-dispatchFan-out to SES/SNS/WebSocket123 days
audit-logImmutable event log → S3 archival630 days

Key Design Decisions

Manual offset commit (enable.auto.commit=false) on all consumers for exactly-once semantics on alert generation. Each service maintains its own consumer group — independent scaling and replay. Dead Letter Queue topics ({topic}.dlq) for failed events after 3 retries; DLQ depth monitored as a Prometheus gauge with PagerDuty alerting at depth > 500.

Kafka Connect — S3 Sink

Kafka Connect S3 Sink ships events from shipment-events to S3 in Parquet format with snappy compression, hourly partitioned:

s3://maersk-datalake/shipment-events/year=2024/month=10/day=15/hour=14/

Microservices Deep Dive

Shipment Core Service — Java 17 + Spring Boot 3

Primary business logic service. Handles the complete shipment lifecycle, SLA rule management, and event sourcing. Exposes REST APIs externally and gRPC internally.

Key capabilities

Booking creation and validation. Status state machine (BOOKED → IN_TRANSIT → AT_PORT → DELIVERED). SLA rule storage (PostgreSQL) with Redis cache (5-min TTL). Event sourcing — every state change published to Kafka.

Tech choices

Spring Data JPA + Hibernate. Spring Kafka with custom ProducerInterceptor for trace ID injection. Micrometer + Prometheus for custom business metrics. Flyway for versioned DB migrations. PgBouncer connection pool (max 200, transaction mode).

Alerting Service — Go 1.21

Highest-throughput service. Purpose-built in Go to avoid JVM GC pauses that caused Java p99 spikes above 500ms. Uses a goroutine worker pool for parallel rule evaluation.

Kafka consumer
   ↓
Event channel (buffered, cap=1000)
   ↓
Worker pool (20 goroutines)
   ↓
Parallel rule checks (goroutines)
   ├── checkDelay()
   ├── checkRouteDeviation()
   ├── checkPortCongestion()
   └── checkSLABreach()
   ↓
Alert → alert-triggers Kafka topic

Rule engine loads config from PostgreSQL at startup, hot-reloads every 5 minutes via background goroutine:

{
  "delay_threshold_hours": 2,
  "route_deviation_km": 50,
  "port_congestion_eta_multiplier": 1.5,
  "sla_warning_buffer_hours": 4
}

Circuit breaker on gRPC calls to core service using sony/gobreaker. Retry with exponential backoff + jitter (max 3 retries).

ETA Prediction Service — Java + Python Sidecar

Hybrid service: Java handles Kafka consumption and orchestration; a Python FastAPI sidecar serves an XGBoost model over local HTTP (POST localhost:8001/predict). Features: vessel speed, historical port wait times, weather conditions, route segment distance. Results published to eta-updates topic.

Notification Service — Go 1.21

Consumes notification-dispatch and fans out to three channels: AWS SES for email (HTML templates via html/template), AWS SNS for SMS on critical SLA breaches, WebSocket push to dashboard via gorilla/websocket hub pattern with per-shipment subscription rooms. Idempotency key per alert prevents duplicate sends.

Storage Layer

AWS RDS — PostgreSQL 15

Multi-AZ deployment. Automated failover <60s. Read replica for reporting and ML feature extraction. PgBouncer transaction-mode pooling (max 200 connections). Key tables: shipments, shipment_events (event store), sla_rules, alerts, audit_log. shipment_events partitioned by month.

Amazon ElastiCache — Redis 7

Cluster mode disabled (3 replicas). Use cases: active shipment state cache (TTL 10m), SLA rule cache (TTL 5m), rate limiting (sliding window), distributed locks via Redisson for idempotent Kafka processing, alert deduplication (SET + EXPIRE, 30s window).

Amazon S3 + AWS Glue (Data Lake)

Kafka Connect S3 Sink. Parquet format, snappy compression, hourly partitioning. Glue crawler auto-discovers schema. Athena for ad-hoc queries and ML dataset extraction. Lifecycle: Standard → IA after 30d → Glacier after 90d. ~10x storage savings vs JSON.

Amazon OpenSearch Service

3-node cluster, dedicated masters. Index: alerts-{yyyy-mm} with monthly rollover. Used for: alert history search (ops dashboard), full-text shipment search, log aggregation from Fluentd. Index aliases enable zero-downtime reindexing during mapping changes.

Infrastructure + Deployment

Kubernetes on AWS EKS

Managed node groups: m5.xlarge for Java services, c5.large for Go services, Spot c5.large for alerting pods (stateless, safely interruptible). Namespaces: shipment-core, alerting, notification, monitoring.

Every service has a PodDisruptionBudget (minAvailable: 1). Config via ConfigMap; secrets pulled from AWS Secrets Manager using External Secrets Operator — no secrets in Git.

Autoscaling Strategy

Java services — CPU-based HPA

Scales at CPU > 70%. Min 3 replicas, max 10. Stateless — no affinity rules needed. Spring Boot actuator /health endpoint used for readiness and liveness probes.

Go alerting — KEDA (Kafka lag)

KEDA scales alerting pods directly on Kafka consumer lag metric. CPU is a lagging indicator for this workload; lag is direct and responsive. Scale-to-zero not used — minimum 2 pods for HA.

Networking — Zero Trust

All services in VPC private subnets. NAT Gateway for outbound. No public IPs on any pod. Service-to-service communication authenticated with mTLS — certificates issued by AWS ACM Private CA with automated rotation. External traffic enters only via API Gateway (JWT auth via Cognito) → ALB → services.

Observability

Prometheus + Alertmanager

Both Java (Micrometer) and Go (prometheus/client_golang) expose /metrics. Prometheus scrapes every 15s. Custom ServiceMonitors in each namespace.

Key alerting rules:

KafkaConsumerLagHigh

Fires if kafka_consumer_group_lag > 10,000 for 2 minutes. Routes → PagerDuty P2.

AlertingLatencyHigh

Fires if p99(alerting_rule_evaluation_duration_seconds) > 100ms for 1 minute. Routes → PagerDuty P1.

DLQDepthCritical

Fires immediately (for: 0m) if alerting_dlq_depth > 500. Routes → PagerDuty P1 + Slack.

SLABreachRateHigh

Fires if rate of SLA breach alerts > 50/min for 5 minutes. Routes → Slack #ops-alerts.

Distributed Tracing — AWS X-Ray + OpenTelemetry

Java services: opentelemetry-java-agent auto-instrumentation. Go services: manual with go.opentelemetry.io/otel. Trace ID injected into Kafka record headers (X-B3-TraceId) for cross-service correlation. X-Ray service map shows full graph: API Gateway → Core Service → Kafka → Alerting Service → RDS.

Structured Logging

Java: Logback + logstash-logback-encoder (JSON output). Go: zerolog (JSON, zero-allocation). Consistent log fields across all services: traceId, shipmentId, service, level, message, durationMs. Fluentd DaemonSet ships to CloudWatch Logs. 30-day retention per log group.

Key Technical Decisions

Go for alerting service
Java GC pauses caused p99 spikes >500ms; Go goroutines eliminated this, achieving <100ms p99 consistently
Kafka over SQS
Need for multiple independent consumer groups and event replay capability — not possible with SQS
Manual Kafka offset commit
Exactly-once alert semantics — auto-commit risked missing events on pod restart mid-batch
Redis distributed locks
Multiple alerting pods on same partition caused duplicate alert generation without Redisson locking
KEDA over CPU-based HPA
CPU is a lagging indicator for Kafka consumers; scaling directly on consumer lag is responsive and accurate
PgBouncer connection pooling
EKS pods × Spring connections easily exceeded RDS max_connections — PgBouncer in transaction mode solved this
Parquet on S3 vs JSON
10x storage savings; Athena query cost reduced by ~85% due to columnar format and predicate pushdown
mTLS service-to-service
Zero-trust within VPC; prevents lateral movement if a pod is compromised — never rely on network perimeter alone
Hot-reload rule engine (Go)
Alerting rules must be changeable without redeployment; background goroutine polls DB every 5m, zero downtime
